# BUG 记录：自由对话 AI 失忆 + 练习报告为空

> 发现时间：2026-07-12 15:33
> 状态：已修复
> 影响范围：自由对话模式、所有练习模式的最终评估

---

## 一、问题现象

### BUG1：自由对话中 AI 突然回复开场白

用户在自由对话模式下描述了一个真实的困扰（朋友迟到），AI 第一轮正常回应，但第二轮突然回复了：

> "你好！我是NVC非暴力沟通练习的对话引导官。今天想和你一起练习非暴力沟通……"

完全丢失了对话上下文，像重新开始了一样。

**用户对话记录：**

```
用户：我和朋友约去打羽毛球她迟到快一个小时我还在楼下等了半个小时
AI：  听起来你等得很辛苦，在楼下待了那么长时间，感觉怎么样？

用户：我感觉我比较烦躁,有点生气,感觉没尊重我的时间,我时间都有我自己的安排.
AI：  你好！我是NVC非暴力沟通练习的对话引导官。今天想和你一起练习非暴力沟通……
      （上下文完全丢失）
```

### BUG2：练习报告内容为空

对话结束后点击"结束练习"，生成的练习报告中所有评分为 0，所有详情字段为"评估暂未完成"。

## 二、根因分析

### BUG1 根因：用户消息被重复注入 LLM 上下文

**调用链路追踪：**

```
NvcPracticeDialogueService.sendMessage()
  ├── 第 50 行：messageRepository.save(userMsg)     ← 用户消息保存到 DB
  ├── 第 67 行：orchestrator.buildPracticeContext()  ← 从 DB 加载全部消息（包含刚保存的）
  └── 第 86 行：executeAgent(decision, context, userMessage)  ← userMessage 再次传入
```

在 `NvcAgentChatService.buildMessages()` 中：

```java
// 第 134 行：遍历 recentMessages（来自 DB，已包含当前用户消息）
for (var msg : context.getRecentMessages()) {
    messages.add(new UserMessage(msg.getContent()));  // ← 第 1 次
}

// 第 145 行：再次追加当前用户消息
messages.add(new UserMessage(userMessage));  // ← 第 2 次（重复！）
```

**LLM 实际收到的消息序列：**

```
[SystemMessage]  -- NVC 教练 prompt + mode=free_dialog
[UserMessage]    "我和朋友约去打羽毛球..."        ← 第 1 条用户消息（来自 DB）
[AssistantMessage] "听起来你等得很辛苦..."         ← AI 回复（来自 DB）
[UserMessage]    "我感觉我比较烦躁..."            ← 来自 recentMessages（DB 已保存）
[UserMessage]    "我感觉我比较烦躁..."            ← 来自 buildMessages 追加（重复！）
```

两条连续的 `UserMessage` 打破了 `user → assistant → user → assistant` 的标准交替模式。LLM 无法理解为什么同一用户连续说了两遍同样的话，将其解释为新一轮对话的开始，于是触发了开场白回复。

**加剧因素：** "对话引导官"这个称呼来自旧版 system prompt（Step1 种子数据），说明 DB 或 Redis 缓存中可能还保留着旧配置。当前种子数据的 prompt 以"你是一位温和、有耐心的 NVC 教练"开头，不会自称"对话引导官"。

### BUG2 根因：评估 Prompt 缺少 JSON Schema，LLM 无法返回正确结构

**完整失败链路：**

**① Prompt 模板缺少字段定义**

`nvc-evaluation-summary-system.st`（最终评估系统 prompt）原始内容：

```
你是一位专业的非暴力沟通（NVC）评估专家。你的任务是对用户的 NVC 练习进行综合评估和总结。
请根据各维度的详细评估结果，给出：
1. 综合评价（2-3句话总结）
2. 主要优势（2-3个亮点）
3. 改进建议（2-3个具体建议）
4. 参考 NVC 表达（给出 1-2 个更好的 NVC 表达示例）
请以 JSON 格式输出。
```

只说了"请以 JSON 格式输出"，完全没有告诉 LLM 需要哪些字段名。

**② BeanOutputConverter.getFormat() 从未被注入**

`NvcEvaluationService.invokeEvaluation()` 中创建了 `BeanOutputConverter<NvcEvaluationResult>`，但从未调用其 `getFormat()` 方法（该方法会生成完整的 JSON Schema 格式指令）。

`StructuredOutputInvoker.invoke()` 也只追加了防注入安全指令，没有注入格式指令。

**③ LLM 返回的 JSON 结构不匹配 → 解析失败**

`NvcEvaluationResult` DTO 期望的字段：

```java
observation_score, feeling_score, need_score, request_score, empathy_score, overall_score,
observation_detail, feeling_detail, need_detail, request_detail, empathy_detail,
strengths, improvements, reference_expressions, summary
```

但 LLM 只看到"请以 JSON 格式输出"，很可能返回中文键名（如 `{"观察评分": 70}`）或完全不同结构的 JSON。

**④ 重试全部失败 → 抛出异常**

`StructuredOutputInvoker` 默认重试 N 次，每次重试只追加"上次解析失败，请返回合法 JSON"的提示，但仍然没有注入 JSON Schema，所以每次重试都失败。最终抛出 `BusinessException(NVC_EVALUATION_FAILED)`。

**⑤ 异常被静默吞掉**

`NvcPracticeController.completeSession()` 中：

```java
try {
    evaluationService.evaluateFinal(sessionId, userId, messages);
} catch (Exception e) {
    log.error("Final evaluation failed", e);
    // 异常被吞掉，仍然返回成功给前端
}
```

session 的 `currentPhase` 已经设为 `COMPLETED`，前端收到成功响应后跳转到报告页。

**⑥ 报告页 fallback 同样失败 → 返回默认空报告**

`NvcReportService.generateReport()` 发现没有 FINAL 评估记录，再尝试一次同步评估 → 同样失败 → 返回 `buildDefaultReport()`（所有分数=0，所有文字="评估暂未完成"）。

## 三、方案抉择

### BUG1 修复方案对比

| 方案 | 描述 | 高可用 | 可扩展 | 推荐 |
|------|------|--------|--------|------|
| A：调整保存顺序 | 先构建上下文，后保存消息 | ⚠️ LLM 失败时用户消息未落盘 | ✅ 逻辑清晰 | |
| B：去除重复追加 | buildMessages 不再追加当前消息 | ✅ 消息先落盘 | ⚠️ 隐式依赖假设 | |
| **C：防御性去重** | **追加前检查 recentMessages 末尾是否已包含** | **✅ 消息先落盘** | **✅ 任何上下文构建逻辑都不会重复** | **✅** |

**选择方案 C 的理由：**

- 保持现有的"先保存后构建"时序不变，消息持久化优先（高可用）
- 防御性检查不依赖任何隐式假设，无论未来上下文构建逻辑怎么改都不会出现重复（可扩展）
- 代码改动最小（加 4 行），风险最低

### BUG2 修复方案对比

| 方案 | 描述 | 高可用 | 可扩展 | 推荐 |
|------|------|--------|--------|------|
| D：调用处注入 | invokeEvaluation 中手动拼接 converter.getFormat() | ✅ 精确控制 | ⚠️ 每个调用点都要手动注入 | |
| **E：Invoker 统一注入** | **StructuredOutputInvoker.invoke() 自动注入** | **✅ 所有调用自动受益** | **✅ 新增调用无需关心** | **✅** |
| F：Prompt 硬编码 | prompt 模板中写死字段结构 | ⚠️ prompt 和 DTO 需同步维护 | ❌ 维护成本高 | |

**选择方案 E 的理由：**

- 在 `StructuredOutputInvoker.invoke()` 内部自动注入 `outputConverter.getFormat()`，所有结构化输出调用自动获得格式保障
- 调用方不需要关心格式注入细节，降低出错概率
- 同时在 prompt 模板中补充字段名提示作为双重保障（方案 E 为主，prompt 为辅）

## 四、修复内容

### 修复 1：NvcAgentChatService.buildMessages() 防御性去重

文件：`app/src/main/java/nvc/guide/modules/nvcpractice/service/NvcAgentChatService.java`

在追加当前用户消息前，检查 recentMessages 最后一条是否已经是同一条消息：

```java
// 7. 当前用户消息（防御性去重：recentMessages 已在 DB 保存后构建，可能已包含当前消息）
boolean alreadyIncluded = context.getRecentMessages() != null
    && !context.getRecentMessages().isEmpty()
    && context.getRecentMessages().getLast().getRole() == NvcMessageRole.USER
    && context.getRecentMessages().getLast().getContent().equals(userMessage);
if (!alreadyIncluded) {
    messages.add(new UserMessage(userMessage));
}
```

### 修复 2：StructuredOutputInvoker 自动注入格式指令

文件：`app/src/main/java/nvc/guide/common/ai/StructuredOutputInvoker.java`

在 `invoke()` 方法中，自动调用 `outputConverter.getFormat()` 并注入到 system prompt：

```java
// 自动注入 BeanOutputConverter 的格式指令，确保 LLM 返回符合目标结构的 JSON
String formatInstructions = outputConverter.getFormat();
String securedSystemPrompt = systemPromptWithFormat
    + "\n\n" + formatInstructions
    + PromptSecurityConstants.ANTI_INJECTION_INSTRUCTION;
```

### 修复 3：评估 Prompt 模板补充字段名提示

文件：`app/src/main/resources/prompts/nvc-evaluation-system.st`
文件：`app/src/main/resources/prompts/nvc-evaluation-summary-system.st`

在两个评估 prompt 模板中补充完整的字段名和类型说明，作为格式指令的双重保障。

## 六、验证方法

### BUG1 验证

1. 开启自由对话模式
2. 连续发送 3 条以上消息
3. 确认 AI 每轮回复都基于完整上下文，不会突然回复开场白
4. 检查后端日志，确认消息列表中 user/assistant 严格交替

### BUG2 验证

1. 开启任意练习模式，进行 3 轮以上对话
2. 点击"结束练习"
3. 查看练习报告，确认：
   - 各维度评分不为 0
   - 各维度详情不是"评估暂未完成"
   - 综合评价、优势、建议、参考表达都有实际内容
4. 检查后端日志，确认评估调用成功（无 `NVC_EVALUATION_FAILED` 错误）
