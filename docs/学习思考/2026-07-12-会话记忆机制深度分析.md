# 会话记忆机制深度分析 — 短期对话记忆的两种实现

> 分析日期：2026-07-12
> 分析范围：RAG 知识库问答助手 + NVC 练习模块 Agent 对话 的短期会话记忆机制

---

## 一、核心结论：两种截然不同的记忆策略

本项目的两个对话模块采用了 **完全不同的短期记忆策略**：

| 维度 | RAG 知识库问答助手 | NVC 练习模块 Agent 对话 |
|------|-------------------|------------------------|
| **记忆来源** | 数据库 `rag_chat_messages` 表 | 数据库 `nvc_practice_message` 表 |
| **记忆注入方式** | 通过 `.messages(history)` 直接注入 LLM 请求 | 通过 `PracticeContext.recentMessages` 手动组装 |
| **记忆窗口** | 最近 10 条已完成消息 | 最近 20 条消息 |
| **记忆载体** | Spring AI 的 `Message` 对象（UserMessage/AssistantMessage） | 自定义 `NvcPracticeMessageEntity` 转换为 Spring AI Message |
| **Spring AI ChatMemory Advisor** | **未启用**（`message-chat-memory-enabled: false`） | **未启用** |
| **Query Rewrite 中的历史** | 有，格式化为 "用户: xxx / 助手: xxx" | 无（RAG 检索不使用历史） |
| **流式消息的临时状态** | `completed=false` 标记，查询时排除 | 无临时状态，保存后立即可用 |

**关键设计决策**：两个模块都 **没有使用** Spring AI 内置的 `MessageChatMemoryAdvisor`，而是各自在业务层手动管理对话历史。这是一个有意为之的选择。

## 二、Spring AI ChatMemory Advisor — 为什么没用？

### 2.1 配置现状

```yaml
# application.yml
app:
  ai:
    advisors:
      message-chat-memory-enabled: false    # ← 关闭
      message-chat-memory-max-messages: 120
```

### 2.2 LlmProviderRegistry 中的 Advisor 构建

```java
// LlmProviderRegistry.buildDefaultAdvisors()
private List<Advisor> buildDefaultAdvisors(String providerId) {
    List<Advisor> advisors = new ArrayList<>();

    // ToolCallAdvisor（工具调用）
    if (config.isToolCallEnabled()) { ... }

    // MessageChatMemoryAdvisor（自动记忆）
    if (config.isMessageChatMemoryEnabled()) {    // ← false，不进入

    }

    // SimpleLoggerAdvisor
    if (config.isSimpleLoggerEnabled()) { ... }

    // SafeGuardAdvisor
    buildSafeGuardAdvisor().ifPresent(advisors::add);

    return advisors;
}
```

### 2.3 为什么不使用 Spring AI ChatMemory？

Spring AI 的 `MessageChatMemoryAdvisor` 有以下特性：

```
┌─────────────────────────────────────────────────────────────┐
│           Spring AI MessageChatMemoryAdvisor                │
│                                                             │
│  工作原理：                                                  │
│  1. 每次调用前，从 ChatMemory 中加载历史消息                  │
│  2. 将历史消息注入到 prompt 的 messages 中                   │
│  3. 调用完成后，将 user message + assistant reply 保存到      │
│     ChatMemory                                              │
│                                                             │
│  默认存储：InMemoryChatMemory（内存）                        │
│  可选存储：JdbcChatMemory / CassandraChatMemory 等          │
│                                                             │
│  问题：                                                      │
│  ├── 内存存储 → 重启丢失，不支持多实例                       │
│  ├── 按 conversationId 隔离 → 需要额外的会话管理逻辑        │
│  ├── 自动保存 → 流式响应时需要特殊处理                       │
│  └── 全局 Advisor → 所有 ChatClient 共享，无法精细控制       │
└─────────────────────────────────────────────────────────────┘
```

**本项目选择手动管理的原因**：

1. **已有消息持久化**：两个模块都需要将消息存入数据库（有业务意义），ChatMemory 会重复存储
2. **流式响应的复杂性**：流式输出时，AI 回复需要累积后一次性保存，ChatMemory 的自动保存时机不合适
3. **精细控制需求**：
   - RAG 聊天需要排除未完成的消息（`completed=false`）
   - 练习模块需要按 Agent 场景标记消息
   - 两个模块的记忆窗口大小不同
4. **Query Rewrite 需要访问历史**：RAG 模块需要将历史格式化后传给 Rewrite Prompt，ChatMemory 不支持这种用法

### 3.2 历史消息加载（loadHistoryMessages）

这是 RAG 聊天记忆的 **核心方法**：

```java
// RagChatSessionService.loadHistoryMessages()
private List<Message> loadHistoryMessages(Long sessionId) {
    // 1. 查询最近 N+1 条已完成的消息（N = maxMessages，默认 10）
    int limit = queryProperties.getHistory().getMaxMessages() + 1;  // 11
    List<RagChatMessageEntity> recent = messageRepository
        .findRecentCompletedBySessionId(sessionId, PageRequest.of(0, limit));

    // 2. 排除最后一条（当前轮的 user 消息）
    //    因为 prepareStreamMessage 已经保存了当前 user 消息（completed=true）
    //    但 AI 回复还没有（completed=false）
    //    所以最后一条 user 消息是"当前轮"的，不应出现在历史中
    List<RagChatMessageEntity> historyMessages = recent.size() <= 1
        ? List.of()
        : recent.subList(1, recent.size());

    // 3. 反转为时间正序（数据库查询是 DESC，需要反转为 ASC）
    // 4. 转换为 Spring AI 的 Message 对象
    return historyMessages.reversed().stream()
        .map(m -> m.getType() == RagChatMessageEntity.MessageType.USER
            ? (Message) new UserMessage(m.getContent())
            : (Message) new AssistantMessage(m.getContent()))
        .toList();
}
```

**关键细节**：

```
数据库中的消息（按 messageOrder DESC）：
┌─────────────────────────────────────────────────┐
│ messageOrder │ type      │ content    │ completed│
├──────────────┼───────────┼────────────┼──────────┤
│ 10           │ USER      │ "什么是观察"│ true     │ ← 当前轮 user 消息
│ 9            │ ASSISTANT │ "感受是..." │ true     │
│ 8            │ USER      │ "什么是感受"│ true     │
│ 7            │ ASSISTANT │ "观察是..." │ true     │
│ 6            │ USER      │ "NVC是什么" │ true     │
│ 5            │ ASSISTANT │ "NVC是..."  │ true     │
│ ...          │           │            │          │
└─────────────────────────────────────────────────┘

查询：findRecentCompletedBySessionId(sessionId, limit=11)
返回：[msg#10, msg#9, msg#8, msg#7, msg#6, msg#5, ...]  (DESC)

排除第一条（当前轮 user 消息）：
historyMessages = [msg#9, msg#8, msg#7, msg#6, msg#5, ...]

反转为正序：
history = [msg#5, msg#6, msg#7, msg#8, msg#9]
         = [UserMessage("NVC是什么"), AssistantMessage("NVC是..."),
            UserMessage("什么是感受"), AssistantMessage("感受是..."),
            UserMessage("什么是观察"), ...]

但注意：这里不包含当前轮的 user 消息（msg#10），
因为 AI 还没有回复，这条消息会被 buildUserPrompt 单独处理。
```

### 3.3 消息的 completed 字段 — 流式响应的临时状态

```java
// RagChatMessageEntity
private Boolean completed = true;  // 默认完成

// prepareStreamMessage() 中：
// 1. 保存用户消息（completed=true，立即完成）
userMessage.setCompleted(true);
messageRepository.save(userMessage);

// 2. 创建 AI 占位消息（completed=false，等待流式完成）
assistantMessage.setCompleted(false);
assistantMessage.setContent("");
messageRepository.save(assistantMessage);

// completeStreamMessage() 中：
// 流式完成后更新
message.setContent(fullContent);
message.setCompleted(true);
messageRepository.save(message);
```

**为什么需要这个字段？**

```
场景：用户发送消息后，流式响应正在进行中，此时另一个请求查询历史。

如果没有 completed 字段：
├── AI 占位消息（content=""）会被加载到历史中
└── 导致 LLM 收到空的 AssistantMessage，影响回答质量

有 completed 字段：
├── findRecentCompletedBySessionId 只查询 completed=true 的消息
└── AI 占位消息不会被加载，直到流式完成并更新为 completed=true
```

### 3.5 历史在 Query Rewrite 中的使用

历史消息不仅用于多轮对话，还用于 **查询改写**，帮助 LLM 理解追问意图：

```java
// KnowledgeBaseQueryService.rewriteQuestion()
private String rewriteQuestion(String question, List<Message> history) {
    Map<String, Object> variables = new HashMap<>();
    variables.put("question", question);
    variables.put("history", formatHistoryForRewrite(history));  // 格式化历史
    String rewritePrompt = rewritePromptTemplate.render(variables);
    String rewritten = getChatClient().prompt()
        .user(rewritePrompt)
        .call()
        .content();
    return rewritten;
}

// 格式化历史为文本摘要
private String formatHistoryForRewrite(List<Message> history) {
    StringBuilder sb = new StringBuilder();
    for (Message msg : history) {
        if (msg instanceof UserMessage) {
            sb.append("用户: ").append(msg.getText()).append("\n");
        } else if (msg instanceof AssistantMessage) {
            String text = msg.getText();
            // 截断过长的助手回复（最多 200 字符）
            if (text.length() > MAX_REWRITE_HISTORY_CHAR) {
                text = text.substring(0, MAX_REWRITE_HISTORY_CHAR) + "...";
            }
            sb.append("助手: ").append(text).append("\n");
        }
    }
    return sb.toString().trim();
}
```

**改写示例**：

```
历史：
  用户: 什么是 NVC？
  助手: NVC 是非暴力沟通（Nonviolent Communication）的缩写...

当前问题：那它的四要素呢？

改写结果：NVC 非暴力沟通的四个核心要素是什么？
（结合历史上下文，"那它的四要素呢" 被改写为更完整的查询）
```

### 4.2 PracticeContext — 练习上下文

```java
@Data
@Builder
public class PracticeContext {
    private NvcPracticeSessionEntity session;         // 会话信息
    private List<NvcPracticeMessageEntity> recentMessages;  // 最近 20 条消息
    private NvcEvaluationEntity lastEvaluation;       // 最新评估
    private int roundCount;                           // 当前轮次
    private String userProfileSummary;                // 用户档案摘要
    private String scenarioDescription;               // 场景描述
    private String ragContext;                        // RAG 知识
    private NvcScenarioEntity scenario;               // 场景实体
}
```

**PracticeContext 是练习模块记忆的载体**。它不是简单的历史消息列表，而是一个 **丰富的上下文对象**，包含：

1. **对话历史**（`recentMessages`）— 短期记忆
2. **用户档案**（`userProfileSummary`）— 长期记忆
3. **场景描述**（`scenarioDescription`）— 当前任务上下文
4. **RAG 知识**（`ragContext`）— 外部知识增强
5. **评估结果**（`lastEvaluation`）— 反馈记忆
6. **轮次信息**（`roundCount`）— 进度感知

### 4.3 消息组装的完整过程

```java
// NvcAgentChatService.buildMessages()
private List<Message> buildMessages(
    NvcAgentConfigEntity config, PracticeContext context,
    String userMessage, Map<String, String> promptVariables) {

    List<Message> messages = new ArrayList<>();

    // ① 基础 System Prompt（从 Agent 配置读取）
    String systemPrompt = config.getSystemPrompt();

    // ② 注入用户档案（长期记忆）
    // 例如：NVC等级: BEGINNER, 沟通背景: 职场, 性格特征: 内向
    if (context.getUserProfileSummary() != null) {
        systemPrompt += "\n\n[用户档案]\n" + context.getUserProfileSummary();
    }

    // ③ 注入场景信息（任务上下文）
    // 例如：职场迟到对话\n你的同事经常迟到，影响了团队进度...
    if (context.getScenarioDescription() != null) {
        systemPrompt += "\n\n[练习场景]\n" + context.getScenarioDescription();
    }

    // ④ 注入 RAG 知识（外部知识增强）
    // 例如：观察是指客观描述事实，不加入评判...
    if (context.getRagContext() != null) {
        systemPrompt += "\n\n[参考资料]\n" + context.getRagContext();
    }

    // ⑤ 注入路由变量（当前状态）
    // 例如：current_step=OBSERVATION, covered_elements=[observation]
    if (promptVariables != null && !promptVariables.isEmpty()) {
        StringBuilder sb = new StringBuilder("\n\n[当前配置]");
        promptVariables.forEach((key, value) ->
            sb.append("\n").append(key).append("=").append(value));
        systemPrompt += sb;
    }

    // ⑥ 加入 SystemMessage
    messages.add(new SystemMessage(systemPrompt));

    // ⑦ 加入对话历史（短期记忆）
    if (context.getRecentMessages() != null) {
        for (var msg : context.getRecentMessages()) {
            if (msg.getRole() == NvcMessageRole.USER) {
                messages.add(new UserMessage(msg.getContent()));
            } else if (msg.getRole() == NvcMessageRole.ASSISTANT) {
                messages.add(new AssistantMessage(msg.getContent()));
            }
        }
    }

    // ⑧ 加入当前用户消息（防御性去重）
    boolean alreadyIncluded = context.getRecentMessages() != null
        && !context.getRecentMessages().isEmpty()
        && context.getRecentMessages().getLast().getRole() == NvcMessageRole.USER
        && context.getRecentMessages().getLast().getContent().equals(userMessage);
    if (!alreadyIncluded) {
        messages.add(new UserMessage(userMessage));
    }

    return messages;
}
```

### 4.4 防御性去重机制

```java
// 防御性去重：recentMessages 已在 DB 保存后构建，可能已包含当前消息
boolean alreadyIncluded = context.getRecentMessages() != null
    && !context.getRecentMessages().isEmpty()
    && context.getRecentMessages().getLast().getRole() == NvcMessageRole.USER
    && context.getRecentMessages().getLast().getContent().equals(userMessage);
if (!alreadyIncluded) {
    messages.add(new UserMessage(userMessage));
}
```

**为什么需要这个？**

```
执行流程：
1. 保存用户消息到 DB  ← 此时 DB 中已有当前 user 消息
2. buildPracticeContext ← 从 DB 加载最近 20 条消息，已包含当前 user 消息
3. buildMessages       ← 如果不检查，会重复添加当前 user 消息

防御性去重：
检查 recentMessages 的最后一条是否就是当前 user 消息
如果是 → 不再添加（避免 LLM 收到重复的 user 消息）
如果不是 → 正常添加
```

## 六、Query Rewrite — RAG 聊天独有的记忆增强

Query Rewrite 是 RAG 聊天助手独有的记忆增强机制，练习模块没有使用。

### 6.1 工作原理

```
用户输入："那它的四要素呢？"
        │
        ▼
┌───────────────────────────────────────────────────┐
│ Query Rewrite                                     │
│                                                   │
│ 输入：                                            │
│ ├── question: "那它的四要素呢？"                   │
│ └── history:                                      │
│     用户: 什么是 NVC？                             │
│     助手: NVC 是非暴力沟通（Nonviolent...          │
│                                                   │
│ Prompt：                                          │
│ "你是一个检索查询改写助手。你的任务是把用户原始    │
│  问题改写成更适合知识库检索的单句查询。"           │
│                                                   │
│ 输出："NVC 非暴力沟通的四个核心要素是什么？"       │
└───────────────────────────────────────────────────┘
        │
        ▼
用改写后的查询进行向量检索（效果更好）
```

### 6.2 候选查询策略

```java
// buildQueryContext()
Set<String> candidates = new LinkedHashSet<>();
candidates.add(rewrittenQuestion);    // 优先使用改写后的查询
candidates.add(normalizedQuestion);   // 回退到原始查询

// retrieveRelevantDocs()
for (String candidateQuery : queryContext.candidateQueries()) {
    List<Document> docs = vectorService.similaritySearch(candidateQuery, ...);
    if (hasEffectiveHit(docs)) {
        return docs;  // 第一个有命中的查询
    }
}
```

**策略**：先尝试改写后的查询，如果没有命中再用原始查询。这样既利用了历史上下文的改写能力，又保证了检索的鲁棒性。

## 七、流式响应中的记忆时序问题

### 7.1 RAG 聊天的时序设计

```
时间线：
t0 ─── 用户发送消息 ────────────────────────────────────────────────
t1 ─── prepareStreamMessage() ─────────────────────────────────────
       │ 保存 user 消息（completed=true）
       │ 创建 AI 占位（completed=false）
t2 ─── loadHistoryMessages() ──────────────────────────────────────
       │ 查询 completed=true 的消息
       │ 排除当前轮 user 消息（因为它是"进行中"的轮次）
       │ 结果：上一轮及之前的历史
t3 ─── 流式输出开始 ───────────────────────────────────────────────
       │
       │  ... 如果此时另一个请求查询历史 ...
       │  loadHistoryMessages() 只会返回 completed=true 的消息
       │  AI 占位消息（completed=false）不会被返回
       │  当前轮 user 消息虽然 completed=true，但会被排除
       │
t4 ─── 流式输出完成 ───────────────────────────────────────────────
       │ completeStreamMessage()
       │ AI 消息更新为 completed=true，content=完整回复
t5 ─── 下一次消息 ─────────────────────────────────────────────────
       │ loadHistoryMessages() 现在会包含 t1 的 user 消息
       │ 和 t4 的 AI 回复
```

### 7.2 练习模块的时序设计

```
时间线：
t0 ─── 用户发送消息 ────────────────────────────────────────────────
t1 ─── 保存 user 消息到 DB ────────────────────────────────────────
       │ sequenceNum = N
t2 ─── buildPracticeContext() ──────────────────────────────────────
       │ 加载最近 20 条消息（包含 t1 的 user 消息）
t3 ─── buildMessages() ─────────────────────────────────────────────
       │ 组装 System + History + Current
       │ 防御性去重：检查 history 最后一条是否 == 当前消息
t4 ─── 流式输出 ───────────────────────────────────────────────────
       │
t5 ─── 流式完成 ───────────────────────────────────────────────────
       │ 保存 AI 回复（sequenceNum = N+1）
```

**关键区别**：练习模块没有 `completed` 字段的设计，因为它在保存 user 消息后 **立即** 构建上下文，不存在"中间状态"的问题。而 RAG 聊天因为有流式输出的延迟，需要 `completed` 字段来区分"进行中"和"已完成"的消息。

## 九、总结：设计哲学

### 9.1 为什么不用 Spring AI ChatMemory？

| 原因 | 说明 |
|------|------|
| **已有持久化** | 两个模块都需要将消息存入数据库（有业务意义），ChatMemory 会重复存储 |
| **流式响应** | 流式输出时需要特殊处理（RAG 用 `completed` 字段，练习用 `doOnComplete` 回调） |
| **精细控制** | 两个模块的记忆窗口、过滤条件、注入方式都不同 |
| **Query Rewrite** | RAG 模块需要将历史格式化后传给 Rewrite Prompt |
| **上下文丰富度** | 练习模块的 PracticeContext 包含远不止历史消息的信息 |

### 9.2 手动管理的优势

```
┌─────────────────────────────────────────────────────────────┐
│                    手动管理记忆的优势                         │
│                                                             │
│ 1. 完全可控                                                 │
│    ├── 窗口大小可独立配置                                   │
│    ├── 过滤条件可独立定义                                   │
│    └── 注入位置可独立控制                                   │
│                                                             │
│ 2. 业务语义丰富                                             │
│    ├── RAG：completed 字段区分流式状态                      │
│    └── 练习：agentScene、step 等业务标记                    │
│                                                             │
│ 3. 复用已有存储                                             │
│    ├── 消息表已经是业务数据的一部分                          │
│    └── 不需要额外的 ChatMemory 存储                         │
│                                                             │
│ 4. 灵活扩展                                                 │
│    ├── RAG：历史可用于 Query Rewrite                        │
│    └── 练习：历史可用于 Agent 调度和评估                    │
│                                                             │
│ 5. 避免 Advisor 冲突                                        │
│    ├── ChatClient 全局共享                                  │
│    └── Advisor 对所有调用生效，无法精细控制                  │
└─────────────────────────────────────────────────────────────┘
```
