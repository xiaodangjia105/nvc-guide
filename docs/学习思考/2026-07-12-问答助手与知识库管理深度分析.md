## 一、整体架构概览

本项目的问答助手基于 **RAG（Retrieval-Augmented Generation）** 架构，核心思想是：**先从知识库中检索相关内容，再将检索结果作为上下文注入 LLM Prompt 生成回答**。

### 1.1 技术栈

| 层次 | 技术选型 | 说明 |
|------|----------|------|
| **向量数据库** | PostgreSQL + pgvector | 使用 Spring AI 的 `PgVectorStore`，HNSW 索引，余弦距离 |
| **Embedding 模型** | 阿里云 DashScope `text-embedding-v3` | 1024 维向量，批量大小限制 10 |
| **LLM 对话模型** | 可配置（DashScope/MiMo/DeepSeek/Kimi/GLM 等） | 通过 `LlmProviderRegistry` 统一管理 |
| **异步任务** | Redis Stream | 向量化任务的生产者-消费者模式 |
| **文件存储** | RustFS（S3 兼容） | 存储原始上传文件 |
| **文档解析** | Apache Tika / 自定义解析器 | 支持 PDF、DOCX、TXT、MD 等格式 |
| **文本分块** | `TokenTextSplitter` | 约 800 tokens/chunk，基于标点边界切分 |

## 二、知识库管理 — 完整实现

### 2.1 数据模型

#### 2.1.1 KnowledgeBaseEntity（知识库实体）

对应数据库表 `knowledge_bases`，核心字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | Long | 主键，自增 |
| `fileHash` | String(64) | 文件内容的 SHA-256 哈希，**唯一索引，用于去重** |
| `name` | String | 知识库名称（用户自定义或从文件名提取） |
| `category` | String(100) | 分类/分组（如 "NVC_THEORY"、"SPEECH_TEMPLATE"） |
| `type` | KnowledgeBaseType | 知识库类型枚举 |
| `originalFilename` | String | 原始文件名 |
| `fileSize` | Long | 文件大小（字节） |
| `contentType` | String | MIME 类型 |
| `storageKey` | String(500) | RustFS 存储的文件 Key |
| `storageUrl` | String(1000) | RustFS 存储的文件 URL |
| `vectorStatus` | VectorStatus | 向量化状态（PENDING/PROCESSING/COMPLETED/FAILED） |
| `vectorError` | String(500) | 向量化失败时的错误信息 |
| `chunkCount` | Integer | 向量分块数量 |
| `accessCount` | Integer | 访问次数 |
| `questionCount` | Integer | 提问次数 |
| `uploadedAt` | LocalDateTime | 上传时间 |
| `lastAccessedAt` | LocalDateTime | 最后访问时间 |

#### 2.1.2 KnowledgeBaseType（知识库类型枚举）

```java
public enum KnowledgeBaseType {
    NVC_THEORY,       // NVC 理论知识（四要素、核心原则等）
    SPEECH_TEMPLATE,  // 话术模板（职场/家庭/亲密关系场景话术）
    EMOTION_VOCAB,    // 情绪词汇（正面/负面感受词汇、需求词汇）
    USER_CASE,        // 用户案例（成功/失败案例对比）
    PERSONAL_WIKI     // 个人 Wiki（按用户隔离）
}
```

#### 2.1.3 VectorStatus（向量化状态枚举）

```java
public enum VectorStatus {
    PENDING,     // 待处理（刚上传/刚入队）
    PROCESSING,  // 处理中（消费者已开始处理）
    COMPLETED,   // 完成（向量化成功）
    FAILED       // 失败（向量化失败，可查看 vectorError）
}
```

#### 2.1.4 RagChatSessionEntity（聊天会话实体）

对应数据库表 `rag_chat_sessions`，通过中间表 `rag_session_knowledge_bases` 与知识库建立 **多对多** 关系：

```java
@Entity
public class RagChatSessionEntity {
    private Long id;
    private String title;                    // 会话标题
    private SessionStatus status;            // ACTIVE / ARCHIVED
    private Boolean isPinned;                // 是否置顶

    @ManyToMany
    @JoinTable(name = "rag_session_knowledge_bases")
    private Set<KnowledgeBaseEntity> knowledgeBases;  // 关联的知识库

    @OneToMany(mappedBy = "session", cascade = CascadeType.ALL, orphanRemoval = true)
    @OrderBy("messageOrder ASC")
    private List<RagChatMessageEntity> messages;      // 消息列表

    private Integer messageCount;            // 消息数量（冗余字段）
    private LocalDateTime createdAt;
    private LocalDateTime updatedAt;
}
```

**关键设计**：一个会话可以关联多个知识库，查询时会从所有关联知识库中检索。

#### 2.2.2 SeedKnowledgeBaseService 启动流程

`SeedKnowledgeBaseService` 监听 `ApplicationReadyEvent`，在应用启动时自动执行：

```
应用启动
  │
  ▼
监听 ApplicationReadyEvent
  │
  ▼
检查是否已有 NVC 类型的知识库（NVC_THEORY / SPEECH_TEMPLATE / EMOTION_VOCAB / USER_CASE）
  │
  ├── 已有数据 → 检查是否有 FAILED 状态的文档
  │               ├── 有失败文档 → 重试向量化（retryFailedVectorization）
  │               └── 全部正常 → 跳过
  │
  └── 无数据 → 执行种子数据初始化（seedKnowledgeDocuments）
                 │
                 ├── 遍历 4 个文件夹（theory/vocabulary/templates/cases）
                 ├── 读取每个 .md 文件内容
                 ├── 计算 SHA-256 哈希（幂等去重）
                 ├── 创建 KnowledgeBaseEntity（状态=PENDING）
                 ├── 保存到数据库
                 └── 发送向量化任务到 Redis Stream
```

**重试机制**：对于向量化失败的种子文档，启动时会自动从 classpath 重新读取内容并重新发送向量化任务。

### 2.4 向量化处理流程

向量化采用 **Redis Stream 生产者-消费者模式** 异步处理：

#### 2.4.1 生产者（VectorizeStreamProducer）

```java
@Component
public class VectorizeStreamProducer extends AbstractStreamProducer<VectorizeTaskPayload> {

    record VectorizeTaskPayload(Long kbId, String content) {}

    // 发送到 Redis Stream
    // Stream Key: AsyncTaskStreamConstants.KB_VECTORIZE_STREAM_KEY
    // 消息字段: kb_id, content, retry_count

    @Override
    protected void onSendFailed(VectorizeTaskPayload payload, String error) {
        // 发送失败时更新状态为 FAILED
        updateVectorStatus(payload.kbId(), VectorStatus.FAILED, error);
    }
}
```

#### 2.4.2 消费者（VectorizeStreamConsumer）

```java
@Component
public class VectorizeStreamConsumer extends AbstractStreamConsumer<VectorizePayload> {

    record VectorizePayload(Long kbId, String content) {}

    // 生命周期钩子：
    // markProcessing  → 更新状态为 PROCESSING
    // processBusiness → 执行向量化（核心逻辑）
    // markCompleted   → 更新状态为 COMPLETED
    // markFailed      → 更新状态为 FAILED
    // retryMessage    → 重新入队（带 retryCount+1）

    @Override
    protected void processBusiness(VectorizePayload payload) {
        vectorService.vectorizeAndStore(payload.kbId(), payload.content());
    }
}
```

#### 2.4.3 向量化核心（KnowledgeBaseVectorService.vectorizeAndStore）

```
收到向量化任务 (kbId, content)
  │
  ▼
① 删除该知识库的旧向量数据（幂等）
   └── VectorRepository.deleteByKnowledgeBaseId(kbId)
       └── SQL: DELETE FROM vector_store WHERE metadata->>'kb_id' = ?
  │
  ▼
② 文本分块（TokenTextSplitter）
   ├── 默认配置：约 800 tokens/chunk
   ├── 基于标点边界切分
   └── 无重叠
  │
  ▼
③ 为每个 chunk 添加 metadata
   └── metadata.put("kb_id", knowledgeBaseId.toString())
  │
  ▼
④ 分批向量化并存储到 PgVectorStore
   ├── 阿里云 DashScope API 限制 batch size ≤ 10
   ├── 每批调用 vectorStore.add(batch)
   │   └── 内部流程：
   │       ├── 调用 Embedding Model（text-embedding-v3）生成 1024 维向量
   │       └── 写入 PostgreSQL vector_store 表
   └── 更新实体的 chunkCount
```

**向量数据库表结构**（Spring AI PgVectorStore 默认表 `vector_store`）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | varchar | 文档 ID |
| `content` | text | 文本内容 |
| `metadata` | json | 元数据（含 `kb_id` 用于过滤） |
| `embedding` | vector(1024) | 1024 维向量 |

索引类型：**HNSW**，距离度量：**COSINE_DISTANCE**

### 3.4 RAG 核心流程（KnowledgeBaseQueryService）

这是问答助手的 **核心引擎**，完整流程如下：

```
输入: knowledgeBaseIds, question, history
  │
  ▼
Step 1: 预处理
  ├── 验证参数（kbIds 非空、question 非空）
  ├── 更新提问计数（KnowledgeBaseCountService）
  └── 清洗 question（trim）
  │
  ▼
Step 2: Query Rewrite（查询改写）
  ├── 如果 rewriteEnabled=true，调用 LLM 改写查询
  ├── Prompt: knowledgebase-query-rewrite.st
  │   └── "把用户原始问题改写成更适合知识库检索的单句查询"
  ├── 支持结合对话历史理解追问意图
  └── 输出：改写后的查询文本
  │
  ▼
Step 3: 构建候选查询列表
  ├── candidateQueries = [rewrittenQuestion, originalQuestion]
  └── 优先使用改写后的查询，如果没有命中则回退到原始查询
  │
  ▼
Step 4: 动态参数解析（resolveSearchParams）
  根据问题长度动态调整检索参数：
  ├── 短查询（≤4 字符）：topK=20, minScore=0.25
  ├── 中等查询（5-12 字符）：topK=12, minScore=0.28
  └── 长查询（>12 字符）：topK=8, minScore=0.28
  │
  ▼
Step 5: 向量检索（retrieveRelevantDocs）
  遍历候选查询列表，使用第一个有效命中：
  │
  ├── 对每个 candidateQuery:
  │   └── KnowledgeBaseVectorService.similaritySearch(query, kbIds, topK, minScore)
  │       │
  │       ▼
  │   构建 SearchRequest：
  │       ├── query: 查询文本
  │       ├── topK: 返回数量
  │       ├── similarityThreshold: 最低相似度阈值
  │       └── filterExpression: "kb_id in ['1', '2', '3']"（按知识库过滤）
  │       │
  │       ▼
  │   调用 vectorStore.similaritySearch(searchRequest)
  │       │
  │       ├── 成功 → 返回结果
  │       └── 失败 → 回退到本地过滤（similaritySearchFallback）
  │                   ├── 扩大 topK（×3）
  │                   ├── 全库检索
  │                   └── 本地按 kb_id 过滤
  │
  ├── 如果某个候选查询有有效命中 → 返回该结果
  └── 所有候选都没有命中 → 返回空列表
  │
  ▼
Step 6: 结果检查
  ├── 无有效命中 → 返回固定话术："抱歉，在选定的知识库中未检索到相关信息..."
  └── 有有效命中 → 继续
  │
  ▼
Step 7: 构建 Prompt
  ├── System Prompt（knowledgebase-query-system.st）
  │   ├── 角色定义：专业的知识库问答助手
  │   ├── 回答原则：准确性优先、完整性保证、结构化表达、多角度分析
  │   ├── 格式规范：Markdown 标题/列表/段落/强调格式
  │   ├── 质量要求：引用来源、逻辑清晰、深度适中
  │   └── 安全指令：PromptSecurityConstants.ANTI_INJECTION_INSTRUCTION（防注入）
  │
  └── User Prompt（knowledgebase-query-user.st）
      ├── {context} → 检索到的文档片段（用 "\n\n---\n\n" 分隔）
      └── {question} → 用户原始问题
  │
  ▼
Step 8: 流式调用 LLM
  ├── 构建 ChatClient prompt
  │   ├── .system(systemPrompt)
  │   ├── .messages(history)  ← 多轮上下文
  │   └── .user(userPrompt)
  ├── .stream().content() → Flux<String>
  └── normalizeStreamOutput → 探测窗口归一化
      ├── 前 120 字符为探测窗口
      ├── 如果探测到"无信息"模板 → 立即输出固定话术并结束
      └── 否则 → 释放缓冲，实时透传后续 chunk
  │
  ▼
输出: Flux<String>（流式 SSE 响应）
```

### 3.5 向量检索的过滤机制

**核心问题**：用户选择了多个知识库（如 kb_id=1,2,3），如何确保只从这些知识库中检索？

**解决方案**：使用 `filterExpression` 进行前置过滤

```java
// KnowledgeBaseVectorService.buildKbFilterExpression()
private String buildKbFilterExpression(List<Long> knowledgeBaseIds) {
    String values = knowledgeBaseIds.stream()
        .filter(Objects::nonNull)
        .map(String::valueOf)
        .map(id -> "'" + id + "'")
        .collect(Collectors.joining(", "));
    return "kb_id in [" + values + "]";
    // 生成: "kb_id in ['1', '2', '3']"
}
```

这个 filter 会被 Spring AI 的 `PgVectorStore` 翻译成 SQL 的 `WHERE metadata->>'kb_id' IN (...)` 条件。

**回退机制**：如果数据库层面的 filter 失败（比如 JSON 路径查询不支持），会回退到本地过滤：
1. 扩大 topK（×3）从全库检索更多结果
2. 在内存中按 `metadata.kb_id` 过滤
3. 截取 topK 个结果返回

### 3.6 多知识库检索的 metadata 设计

每个文档 chunk 在向量化时都会添加 metadata：

```java
chunks.forEach(chunk -> chunk.getMetadata().put("kb_id", knowledgeBaseId.toString()));
```

**注意**：`kb_id` 存储为 **String 类型**，查询时也使用 String 比较：

```java
// 向量检索过滤
builder.filterExpression("kb_id in ['1', '2', '3']")

// 本地过滤回退
Long kbIdLong = Long.parseLong(kbId.toString());
return knowledgeBaseIds.contains(kbIdLong);
```

## 八、关键设计决策总结

| 决策 | 选择 | 原因 |
|------|------|------|
| 向量数据库 | PostgreSQL + pgvector | 项目已有 PostgreSQL，减少运维复杂度 |
| 向量化异步 | Redis Stream 生产者-消费者 | 解耦上传和向量化，支持重试和失败恢复 |
| 文本分块 | TokenTextSplitter 800 tokens | 平衡检索精度和上下文完整性 |
| 多知识库过滤 | filterExpression + 本地回退 | 优先数据库过滤（高效），失败时回退（可靠） |
| 查询改写 | LLM 改写 + 候选列表 | 提升短查询和模糊查询的检索效果 |
| 动态 topK/minScore | 按问题长度分档 | 短查询需要更宽松的检索，长查询可以更精确 |
| 流式输出探测窗口 | 前 120 字符 | 快速识别"无信息"模板，避免长篇拒答 |
| Prompt 安全 | 防注入指令 + 数据标记 | 防止用户通过问题注入恶意指令 |
| 多轮上下文 | 最近 10 条消息 | 支持追问，但不会因历史过长而影响检索质量 |
| 个性化加权 | 薄弱要素关键词 ×1.5 | 在练习场景中优先展示用户需要加强的知识 |
