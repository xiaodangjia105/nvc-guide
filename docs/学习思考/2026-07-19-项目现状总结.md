# NVC 非暴力沟通练习助手 — 项目现状总结

> 更新时间：2026-07-19
> 阶段：Phase 1（RAG + Agent 工具调用 + Skill 标准化）已基本完成

---

## 一、项目概述

**一句话定位**：基于 AI Agent 的非暴力沟通（NVC）智能练习平台，用户通过文字/语音与多角色 AI 对练，系统实时评估 NVC 四要素表达质量并给出个性化改进建议。

**技术栈**：

| 层级 | 技术选型 |
|------|---------|
| 后端框架 | Spring Boot 4.0 + Java 21 |
| AI 集成 | Spring AI 2.0（OpenAI 兼容模式）+ spring-ai-agent-utils |
| 数据库 | PostgreSQL + pgvector（向量检索） |
| 缓存 | Redis（Redisson 4.0） |
| 对象存储 | RustFS（S3 兼容） |
| 前端 | React 18 + TypeScript + Vite + Tailwind CSS 4 |
| 语音 | 阿里云 DashScope SDK（ASR/TTS）+ WebSocket |
| 图表 | Recharts（雷达图、趋势图） |
| PDF 导出 | iText 8 |
| 文档解析 | Apache Tika |

## 二、核心功能清单

### 2.1 NVC 练习核心模块（`modules/nvcpractice`）

**功能名称**：三种练习模式 + 多 Agent 编排 + 实时评估

**核心能力**：
- **场景驱动模式**（SCENARIO）：AI 生成 NVC 练习场景 → 扮演困难搭档 → 用户练习 → 实时评估
- **自由对话模式**（FREE_DIALOG）：用户自由表达 → AI 引导对话 → 实时评估
- **结构化四步模式**（STRUCTURED_FOUR_STEP）：观察 → 感受 → 需求 → 请求，逐步推进，每步有专属教练
- **多 Agent 编排**：10 种专职 Agent 场景（`NvcAgentScene`），通过 `ModeRouter` 策略模式动态路由
- **实时评估**：LLM 结构化输出，评估观察/感受/需求/请求/共情 5 个维度，每维度 0-100 分
- **流式对话**：SSE 流式输出，支持 metadata/message/done 三种事件类型
- **练习报告**：生成综合评估报告，支持 PDF 导出
- **场景推荐**：基于用户能力雷达的薄弱维度，加权推荐最相关练习场景

**关键类/文件**：

| 文件 | 职责 |
|------|------|
| `NvcAgentOrchestrator.java` | 调度中心：组装 PracticeContext → 决策 Agent → 执行对话 → 反思 |
| `NvcAgentChatService.java` | 对话服务：构建消息列表、注入上下文、调用 LLM、支持工具调用 |
| `NvcPracticeDialogueService.java` | 对话流程编排：保存消息 → 构建上下文 → Agent 调度 → 流式/非流式输出 |
| `NvcEvaluationService.java` | 评估服务：实时评估 + 最终评估，使用 StructuredOutputInvoker |
| `NvcStructuredPracticeService.java` | 结构化四步服务：步骤推进、阈值判断、重置 |
| `NvcScenarioRecommendService.java` | 场景推荐：基于能力雷达的加权排序 |
| `NvcReportService.java` | 报告服务：生成练习报告 + PDF 导出 |
| `NvcRagService.java` | RAG 检索：多知识库类型过滤 + 个性化加权 |
| `NvcSummaryService.java` | 摘要服务：异步更新练习摘要 |
| `ModeRouter.java` | 路由策略接口 |
| `FreeDialogRouter.java` | 自由对话路由实现 |
| `ScenarioRouter.java` | 场景驱动路由实现 |
| `StructuredRouter.java` | 结构化四步路由实现 |

**实现状态**：**已完成** — 三种模式全部可用，10 种 Agent 场景已配置，实时评估 + 最终评估 + 报告生成均已实现。

### 2.3 RAG 知识库系统（`modules/knowledgebase`）

**功能名称**：多知识库管理 + 向量化 + 语义检索 + 种子数据

**核心能力**：
- **多知识库类型**：NVC 理论（theory）、话术模板（templates）、情绪词汇（vocabulary）、用户案例（cases）、个人 Wiki（personal）
- **文档上传与解析**：支持 Markdown / PDF / DOCX / TXT，使用 Apache Tika 解析
- **向量化存储**：文本分块（TokenTextSplitter）→ Embedding（DashScope text-embedding-v3）→ pgvector 存储
- **语义检索**：pgvector COSINE 相似度搜索，支持多知识库 ID 过滤
- **查询重写**：LLM 对用户查询进行重写，提高检索质量
- **RAG 对话**：检索增强生成，流式输出
- **种子数据**：应用启动时自动扫描 `classpath:knowledge/` 目录，注册 15 份 NVC 知识文档并自动向量化
- **向量化重试**：自动检测向量化失败的文档并重试

**关键类/文件**：

| 文件 | 职责 |
|------|------|
| `KnowledgeBaseController.java` | 知识库管理 API（CRUD + 上传） |
| `RagChatController.java` | RAG 对话 API |
| `KnowledgeBaseVectorService.java` | 向量存储服务（分块 + 向量化 + 检索） |
| `KnowledgeBaseQueryService.java` | 查询服务（查询重写 + 混合检索 + 流式输出） |
| `KnowledgeBaseUploadService.java` | 上传服务（Tika 解析 + 存储） |
| `SeedKnowledgeBaseService.java` | 种子数据服务（启动时自动初始化） |
| `VectorizeStreamProducer/Consumer.java` | 异步向量化（Redis Stream） |

**知识库内容（15 份种子文档）**：

| 类型 | 文档 |
|------|------|
| NVC 理论 | 观察与评价的区别、感受的识别与表达、需求的探索与理解、请求的表达与沟通、NVC 四要素综合运用 |
| 情绪词汇 | 正面感受词汇库、负面感受词汇库、需求词汇库 |
| 话术模板 | 职场沟通话术、家庭关系话术、亲密关系话术、社交场景话术 |
| 用户案例 | 成功案例集、失败案例对比、常见场景改写 |

**实现状态**：**已完成** — 知识库管理、向量化、语义检索、种子数据、RAG 对话均已实现。

## 三、已实现的链路

### 3.1 文字练习端到端链路

```
用户创建练习会话
  ↓
选择练习模式（场景驱动 / 自由对话 / 结构化四步）
  ↓
用户发送消息 → NvcPracticeDialogueService.sendMessage()
  ↓
保存用户消息到 DB
  ↓
NvcAgentOrchestrator.buildPracticeContext()
  ├── 加载会话信息 + 最近 20 条消息
  ├── 加载最新评估结果
  ├── 加载用户档案摘要（NvcProfileService）
  ├── 加载场景描述（如有）
  └── RAG 知识检索（NvcRagService.retrieve/retrievePersonalized）
       ├── 按知识库类型过滤（NVC_THEORY / SPEECH_TEMPLATE / EMOTION_VOCAB）
       └── 个性化加权（薄弱要素 × 1.5）
  ↓
NvcAgentOrchestrator.decideNextAgent()
  ├── ModeRouter 策略路由（FreeDialogRouter / ScenarioRouter / StructuredRouter）
  └── 返回 AgentDecision（scene + reason + tools）
  ↓
NvcAgentOrchestrator.executeAgent()
  ├── NvcAgentChatService.chat/chatStream()
  │   ├── 构建消息列表（系统提示词 + 用户档案 + 场景 + RAG 知识 + 对话历史）
  │   ├── 注入工具回调（NvcToolRegistry.toFunctionCallbacks）
  │   └── 调用 LLM（支持 Function Calling）
  └── Agent 可调用工具（rag_search / profile_query / evaluate_nvc 等）
  ↓
保存 AI 回复到 DB
  ↓
异步触发摘要更新（NvcSummaryService）
  ↓
结构化四步模式：检查步骤推进（NvcStructuredPracticeService）
  ↓
返回 DialogueResponse（reply + scene + reason + action + step）
```

### 3.2 评估反馈链路

```
用户发送消息 → AI 回复完成
  ↓
NvcEvaluateStreamProducer（Redis Stream）
  ↓
NvcEvaluateStreamConsumer 异步消费
  ↓
NvcEvaluationService.evaluateRealtime()
  ├── 加载 Prompt 模板（prompts/nvc-evaluation-system.st）
  ├── 构建评估输入（用户消息 + AI 上下文 + 当前步骤）
  ├── StructuredOutputInvoker 调用 LLM
  │   └── BeanOutputConverter<NvcEvaluationResult> 结构化输出
  └── 保存评估结果到 DB（观察/感受/需求/请求/共情 5 维度分数）
  ↓
前端实时展示评估卡片（NvcEvaluationCard）
  ↓
练习结束 → NvcEvaluationService.evaluateFinal()（综合评估）
  ↓
NvcReportService.generateReport()（生成练习报告）
  ↓
NvcProfileService 更新用户能力分数
```

### 3.3 场景生成 → 练习 → 评估 → 推荐链路

```
用户请求生成场景 / 系统推荐场景
  ↓
NvcScenarioService.generateScenario()（AI 生成）
  或 NvcScenarioRecommendService.recommend()（基于能力雷达推荐）
  ↓
创建练习会话（绑定场景 ID）
  ↓
ScenarioRouter 路由：
  ├── CREATED 阶段 → SCENARIO_GENERATOR（引入场景）
  └── IN_PROGRESS 阶段 → DIFFICULT_PARTNER（困难搭档角色扮演）
  ↓
用户与困难搭档对话练习
  ↓
实时评估 → NvcEvaluationService.evaluateRealtime()
  ↓
练习结束 → 最终评估 → 更新能力分数
  ↓
能力分数变化 → 影响下次场景推荐权重
  ↓
薄弱维度分数越低 → 权重越高 → 优先推荐相关场景
```

### 3.4 RAG 检索 → 知识注入 → Agent 回复链路

```
用户发送消息
  ↓
NvcAgentOrchestrator.buildPracticeContext()
  ↓
buildRagQuery() 构建检索查询
  ├── 优先级 1：场景描述（场景驱动模式）
  ├── 优先级 2：用户最新消息（自由对话模式）
  └── 优先级 3：练习模式 + 当前步骤（结构化四步模式）
  ↓
findWeakElementKeyword() 查找用户薄弱要素
  ↓
NvcRagService.retrieve/retrievePersonalized()
  ├── 查找匹配的知识库 ID（按类型过滤）
  ├── KnowledgeBaseVectorService.similaritySearch()
  │   ├── pgvector COSINE 相似度搜索
  │   └── 支持多知识库 ID 过滤
  └── 个性化加权排序（薄弱要素关键词匹配 × 1.5）
  ↓
NvcRagService.formatForPrompt() 格式化为 Prompt 注入文本
  ↓
NvcAgentChatService.buildMessages() 注入系统提示词
  ├── [用户档案] 用户 NVC 等级、沟通风格、性格特征
  ├── [练习场景] 场景标题 + 描述
  ├── [参考资料] RAG 检索结果
  └── [当前配置] 路由变量（mode、covered_elements）
  ↓
LLM 生成回复（基于增强后的上下文）
```

### 3.5 语音练习端到端链路

```
用户进入语音练习页面
  ↓
建立 WebSocket 连接（NvcVoiceWebSocketHandler）
  ↓
启动 ASR 会话（AsrProvider.startSession）
  ↓
用户说话 → 音频数据通过 WebSocket 传输
  ↓
VoicePipelineCoordinator.handleAudioData()
  ├── 回声消除：AI 说话时丢弃
  └── ASR 识别：QwenAsrProvider 实时识别
       ├── onPartial → 实时字幕（用户侧）
       └── onFinal → UtteranceMergeBuffer 合并
  ↓
VAD 检测静音 / 用户手动提交
  ↓
VoicePipelineCoordinator.handleSubmit()
  ├── CAS 防并发（processing.compareAndSet）
  ├── 标记 AI 说话中（markAiSpeaking）
  ├── 构建 PracticeContext + Agent 调度
  ├── 构建 Prompt（NvcVoicePromptService）
  ├── LLM 流式调用（NvcVoiceLlmService.chatStreamSentences）
  │   ├── onToken → 实时字幕（AI 侧）
  │   └── onSentence → 触发分片 TTS
  │       └── CompletableFuture 异步合成 → 发送音频块
  ├── 保存消息到 DB
  ├── 发送最终文本
  └── 触发异步评估（NvcVoiceEvaluationService）
  ↓
标记 AI 静默（markAiSilent）
  ↓
用户继续说话 → 循环
```

```

  ↓

  ├── 存储到 RustFS（S3 兼容）
```

## 四、技术架构亮点

### 4.1 多 Agent 编排架构（ModeRouter 策略模式）

**设计思路**：不同练习模式需要不同的对话策略，单个 Agent 难以同时胜任所有角色。

**架构设计**：
- `NvcAgentOrchestrator` 作为调度中心，负责组装 PracticeContext、决策 Agent、执行对话
- `ModeRouter` 策略接口，每种练习模式有独立的路由实现
- `EnumMap<NvcPracticeMode, ModeRouter>` 动态路由，运行时根据模式选择路由
- 10 种 `NvcAgentScene` 枚举，每种场景有独立的系统提示词和工具配置

**技术亮点**：
- 策略模式解耦不同模式的路由逻辑，新增模式只需实现 `ModeRouter` 接口
- Agent 配置存储在 DB 中，支持热更新（`NvcAgentConfigService`）
- 路由决策基于对话状态（轮次、阶段、评估结果），而非简单关键词匹配

### 4.2 Agent 工具调用框架（NvcTool + NvcToolRegistry）

**设计思路**：Agent 需要调用外部能力（RAG 检索、用户档案查询、评估等），需要一个标准化的工具框架。

**架构设计**：
- `NvcTool` 接口定义工具契约（name / description / inputSchema / execute）
- `NvcToolRegistry` 使用 Spring 自动注入收集所有 `NvcTool` 实现
- `NvcToolSceneMapping` 定义每个 Agent 场景可用的工具子集
- `NvcToolContext` 通过 Spring AI ToolContext 注入 userId / sessionId / practiceContext

**技术亮点**：
- 实现 `NvcTool` 接口 + `@Component` 注解即可自动注册，零配置
- `NvcToolRegistry` 适配 Spring AI `FunctionToolCallback`，无缝集成 Function Calling
- 场景-工具映射表控制每个 Agent 的工具权限，避免工具滥用
- 工具执行有统一的错误处理（`NvcToolResult`），Agent 收到错误后用自然语言回复用户

### 4.3 RAG 个性化检索

**设计思路**：通用 RAG 检索对所有用户返回相同结果，无法针对性地帮助用户提升薄弱环节。

**架构设计**：
- `NvcRagService` 封装向量检索逻辑，支持多知识库类型过滤
- `retrievePersonalized()` 方法基于用户薄弱要素加权排序
- 加权算法：薄弱要素关键词匹配的内容得分 × 1.5
- 薄弱要素从用户能力雷达中自动提取（最低分维度）

**技术亮点**：
- 个性化检索让 Agent 的回复更有针对性，比如用户"请求"维度分数低，RAG 会优先检索"请求表达"相关知识
- 多知识库类型过滤（NVC_THEORY / SPEECH_TEMPLATE / EMOTION_VOCAB），避免不相关知识干扰
- 检索结果格式化为 Prompt 注入文本，直接增强 Agent 的上下文

### 4.4 语音管线编排（VoicePipelineCoordinator）

**设计思路**：语音练习涉及 ASR → 合并 → LLM → TTS 多个环节，需要协调处理。

**架构设计**：
- `VoicePipelineCoordinator` 编排完整语音处理流程
- `SessionState` 管理会话状态（AI 说话标记、处理锁、合并缓冲区）
- `UtteranceMergeBuffer` 合并 ASR 片段为完整语句
- `OrderedTtsChunkEmitter` 保证 TTS 分片顺序播放
- 虚拟线程池（`Executors.newVirtualThreadPerTaskExecutor()`）处理阻塞任务

**技术亮点**：
- 回声消除：AI 说话时自动丢弃用户音频，避免自问自答
- CAS 防并发：`processing.compareAndSet(false, true)` 避免重复处理
- 分片 TTS：每句触发 TTS 合成，降低首字延迟
- ASR 自动重连：识别失败后自动重启 ASR 会话

### 4.5 结构化输出 + 重试修复

**设计思路**：LLM 输出的 JSON 可能格式错误，需要重试和自动修复机制。

**架构设计**：
- `StructuredOutputInvoker` 封装结构化输出调用逻辑
- `BeanOutputConverter<T>` 将 LLM 输出转换为 Java 对象
- 重试机制：最多 3 次重试
- 自动修复：JSON 格式错误时尝试提取 JSON 子串

**技术亮点**：
- 评估结果（`NvcEvaluationResult`）使用结构化输出，确保 5 维度分数 + 详情文本的完整解析
- 重试 + 修复机制提高了 LLM 输出的可靠性
- 统一封装，所有需要结构化输出的场景复用

### 4.6 异步处理（Redis Stream）

**设计思路**：评估、向量化等耗时操作不应阻塞主对话流程。

**架构设计**：
- `AbstractStreamProducer/Consumer` 基于 Redis Stream 的生产者/消费者基类
- 评估任务：`NvcEvaluateStreamProducer` → `NvcEvaluateStreamConsumer`
- 向量化任务：`VectorizeStreamProducer` → `VectorizeStreamConsumer`

**技术亮点**：
- 异步解耦：对话流程不等待评估完成，用户立即看到 AI 回复
- Redis Stream 保证消息不丢失，支持消费者组
- 生产者/消费者基类抽象，新增异步任务只需继承并实现业务逻辑

## 五、当前不足 / 待完善

### 5.1 功能层面

| 不足项 | 说明 | 影响 |
|--------|------|------|
| Wiki 模块为骨架 | `WikiWriteTool` 仅有基础实现，缺少完整的 Wiki CRUD 和 Markdown 编辑器 | 个人知识沉淀功能不完整 |
| 主 Agent 未实现 | `NvcToolSceneMapping.getAllTools()` 预留了全量工具，但主 Agent（全能对话入口）尚未实现 | 用户无法通过自然语言完成所有操作 |
| 语音评估较简单 | 语音评估复用文字评估的 Prompt，未针对语音场景优化 | 语音练习的评估质量可能不如文字练习 |
| 缺少用户认证 | 没有登录/注册功能，使用硬编码 userId | 无法多用户使用 |
| 缺少单元测试 | 大部分 Service 没有单元测试 | 代码质量保障不足 |
| 前端状态管理简单 | 使用 React state + props 传递，没有全局状态管理 | 复杂交互可能出现状态不一致 |

### 5.2 技术层面

| 不足项 | 说明 | 影响 |
|--------|------|------|
| 向量检索性能 | pgvector 未配置 HNSW 索引，数据量大时检索变慢 | 知识库内容增多后性能下降 |
| LLM 调用无缓存 | 相同查询每次都调用 LLM，无语义缓存 | 成本高、延迟大 |
| 错误处理不够精细 | 部分 catch 块只 log.warn，没有用户友好的错误提示 | 用户体验不佳 |
| 配置管理分散 | 部分配置在 application.yml，部分在 DB，部分硬编码 | 运维复杂 |
| 缺少监控 | 没有 Prometheus/Grafana 监控，无法观测 LLM 调用延迟、错误率 | 问题排查困难 |
| 前端无 SSR | 纯 SPA，首屏加载慢 | SEO 不友好、首屏体验差 |

### 5.3 代码层面

| 不足项 | 说明 | 影响 |
|--------|------|------|
| 部分 Prompt 硬编码 | 部分 Prompt 写在 Java 代码中，未抽取为模板文件 | 修改 Prompt 需要重新编译 |
| 枚举命名不一致 | `NvcAgentScene` 使用全大写下划线，`NvcPracticeMode` 使用全大写 | 代码风格不统一 |
| 部分 DTO 未使用 Record | 一些 DTO 使用 Lombok @Data，一些使用 Java Record | 风格不统一 |
| 缺少 API 版本控制 | API 路径没有版本号（如 `/api/v1/`） | 未来 breaking change 难以管理 |

## 七、总结

本项目是一个功能完整的 NVC 非暴力沟通智能练习平台，核心技术亮点包括：

1. **多 Agent 编排架构**：ModeRouter 策略模式 + 10 种专职 Agent 场景
2. **Agent 工具调用框架**：NvcTool 接口 + NvcToolRegistry 自动注册 + 10 种工具
3. **RAG 个性化检索**：多知识库类型过滤 + 薄弱要素加权
4. **语音管线编排**：ASR → 合并 → LLM → TTS 完整流程
5. **结构化输出 + 重试修复**：评估结果的可靠解析
6. **异步处理**：Redis Stream 解耦耗时操作

项目已覆盖文字练习、语音练习、知识库管理、用户档案、场景库等核心功能，前端 12 个页面 + 11 个组件，后端 6 个业务模块 + 20 个 Entity + 11 个枚举。整体完成度较高，适合作为 AI 应用开发的简历项目。
