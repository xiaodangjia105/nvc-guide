## 2. 技术亮点详解

### 亮点 1：多 Agent 编排架构

**面试话术：**

> "我设计了一个基于策略模式的多 Agent 编排架构。核心是 `NvcAgentOrchestrator`，它使用 `ModeRouter` 接口为不同的练习模式（场景驱动、自由对话、结构化四步）提供独立的路由逻辑。
>
> 每种模式下有多个专职 Agent，比如场景驱动模式下有 `SCENARIO_GENERATOR`（生成场景）、`DIALOGUE_GUIDE`（对话引导）、`DIFFICULT_PARTNER`（困难搭档）、`NVC_EXPRESSION_EVALUATOR`（评估官）等。
>
> 路由决策基于对话状态（轮次、当前阶段、用户表现），而不是简单的关键词匹配。这样每种练习模式都有最适合的 Agent 序列。"

**技术要点：**
- ModeRouter 策略模式（`FreeDialogRouter` / `ScenarioRouter` / `StructuredRouter`）
- EnumMap<NvcPracticeMode, ModeRouter> 动态路由
- Agent 配置热更新（DB + Redis 缓存）
- 10 种 Agent 场景枚举（`NvcAgentScene`）

---

### 亮点 2：Agent 工具调用框架

**面试话术：**

> "我基于 Spring AI 的 Function Calling 机制实现了一个 Agent 工具调用框架。核心设计是 `NvcTool` 接口，每个工具定义名称、描述、输入 Schema 和执行方法。
>
> `NvcToolRegistry` 使用 Spring 的自动注入收集所有 `NvcTool` 实现，然后转换为 Spring AI 的 `FunctionCallback` 列表。Agent 在对话时可以动态调用这些工具。
>
> 比如用户说'帮我看看最近的练习情况'，Agent 会调用 `DashboardQueryTool` 获取数据后再回复。整个过程对用户是透明的。"

**技术要点：**
- `NvcTool` 接口设计（name / description / inputSchema / execute）
- `NvcToolRegistry` 自动注册（Spring `List<NvcTool>` 注入）
- Spring AI `FunctionCallback` 适配
- 10 种工具实现（RAG、Wiki、档案、评估、场景、仪表盘等）

---

### 亮点 3：RAG 知识检索系统

**面试话术：**

> "我实现了一个多知识库的 RAG 检索系统。知识库按类型分为 NVC 理论、话术模板、情绪词汇、用户案例和个人 Wiki。
>
> 检索策略是混合检索——向量相似度（pgvector COSINE）和关键词检索（PostgreSQL full-text search）结合，权重 7:3。
>
> 个性化方面，系统会根据用户的薄弱环节（比如'请求'维度分数低）加权检索相关内容，让 Agent 的回复更有针对性。"

**技术要点：**
- pgvector COSINE 相似度检索
- PostgreSQL full-text search 关键词检索
- 混合排序算法（向量 × 0.7 + 关键词 × 0.3）
- 个性化权重（薄弱要素 × 1.5）
- 多知识库隔离（type 字段 + 逻辑隔离）

### 亮点 4：个人知识 Wiki 系统

**面试话术：**

> "我设计了一个三层存储的个人知识 Wiki：文件系统存储 Markdown 原文（方便手动编辑），数据库存储元数据，pgvector 存储向量化内容（支持语义检索）。
>
> Wiki 有三种来源：练习结束后 AI 自动生成学习笔记、用户手动记录、以及通过主 Agent 对话时 AI 辅助生成。
>
> 自动沉淀的流程是：练习结束 → AI 分析对话 → 生成 Markdown 笔记 → 写入文件 → 向量化。这样用户的每次练习都会积累成可检索的个人知识。"

**技术要点：**
- 三层存储：文件系统 + DB + pgvector
- AI 自动生成学习笔记（Prompt Engineering）
- Markdown 编辑器（前端）
- 语义检索（基于 pgvector）
- 按用户隔离（userId 过滤）

---

### 亮点 5：可扩展 Skill 体系

**面试话术：**

> "我设计了一个 Skill 标准化框架。每个 Skill 实现 `NvcSkill<T>` 接口，定义名称、描述和执行方法。
>
> Skill 和 Tool 是组合关系——`EvaluateNvcTool` 内部调用 `NvcEvaluationSkill`。这样 Skill 既可以直接调用，也可以通过 Agent 的 Function Calling 间接调用。
>
> 目前有三个 Skill：NVC 评估、场景生成、对话引导。框架设计上支持无感扩展——只需要实现接口并加上 `@Component` 注解就会自动注册。"

**技术要点：**
- `NvcSkill<T>` 泛型接口
- Skill + Tool 组合模式
- Spring 自动注册（`@Component`）
- 结构化输出（`BeanOutputConverter`）

---

### 亮点 6：主 Agent 全能对话入口

**面试话术：**

> "我实现了一个主 Agent 全能对话入口，用户可以通过自然语言完成所有操作。它基于思考链（CoT）引擎工作：理解意图 → 选择工具 → 执行工具 → 生成回复。
>
> 比如用户说'帮我生成一个职场沟通场景'，主 Agent 会调用 `ScenarioGenerateTool` 生成场景，然后用自然语言回复用户。用户说'记录一下我今天的学习感悟'，主 Agent 会调用 `WikiWriteTool` 保存到个人 Wiki。
>
> 前端会展示工具调用过程，让用户知道 AI 在做什么。"

**技术要点：**
- CoT 思考链引擎（Prompt Engineering）
- Function Calling 动态工具选择
- 流式 SSE 对话
- 工具调用可视化（前端）

## 3. 面试常见问题准备

### Q: 为什么用多 Agent 而不是单个 Agent？

> "不同练习模式需要不同的对话策略。场景驱动需要 Agent 扮演'困难搭档'，自由对话需要 Agent 做'共情教练'，结构化四步需要每步有专属'步骤教练'。单个 Agent 很难同时做好这些角色。
>
> 多 Agent 编排让每个 Agent 专注于一件事，通过 Orchestator 协调，既保证了专业性，又保持了灵活性。"

### Q: RAG 在这个场景的价值是什么？LLM 不是已经懂 NVC 了吗？

> "LLM 懂 NVC 理论，但不懂用户的个人知识。我的 RAG 系统主要检索两类内容：
> 1. 话术模板——100+ 条针对不同场景的 NVC 话术，LLM 记不住全部
> 2. 个人 Wiki——用户的练习笔记、真实场景、学习感悟，这些是 LLM 完全不知道的
>
> 所以 RAG 的价值不在于补充通用知识，而在于个性化和专业化。"

### Q: 工具调用的错误处理怎么做？

> "每个工具执行都有 try-catch，返回 `NvcToolResult` 包含成功/失败状态和错误信息。Agent 收到错误后会用自然语言告诉用户出了什么问题，而不是直接报错。
>
> 工具调用还有超时控制（默认 10 秒），避免单个工具卡住整个对话。"

### Q: Skill 和 Tool 的区别是什么？

> "Tool 是 Agent 可以调用的外部能力，面向 LLM 的 Function Calling。Skill 是业务逻辑的封装，面向代码复用。
>
> 一个 Skill 可以被包装为 Tool（比如 `NvcEvaluationSkill` 被包装为 `EvaluateNvcTool`），也可以直接在 Service 中调用。Tool 是 Skill 的一种暴露方式。"

### Q: 向量检索的性能怎么保证？

> "pgvector 支持 IVFFlat 和 HNSW 索引。数据量小的时候用 IVFFlat，数据量大了切 HNSW。
>
> 检索时先用向量相似度粗筛 Top-20，再用关键词精排 Top-5。这样既保证了召回率，又控制了延迟。"
