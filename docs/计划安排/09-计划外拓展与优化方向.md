### 1.1 Token 用量追踪与成本控制 ⭐⭐⭐

#### 问题

每次 LLM 调用的 Token 消耗没有记录，无法回答：
- 每个用户每天消耗多少 Token？
- 哪个 Agent 场景最烧钱？
- 一次练习的平均成本是多少？

#### 方案

**新增实体**：

```java
@Entity
@Table(name = "nvc_token_usage")
public class NvcTokenUsageEntity {
    @Id @GeneratedValue
    private Long id;
    private Long userId;
    private Long sessionId;
    @Enumerated(EnumType.STRING)
    private NvcAgentScene agentScene;
    private String model;            // "qwen-plus"
    private Integer promptTokens;    // 输入 Token
    private Integer completionTokens; // 输出 Token
    private Integer totalTokens;     // prompt + completion
    private BigDecimal cost;         // 费用（元）
    private Long latencyMs;          // 响应延迟
    private Boolean toolCallTriggered; // 是否触发工具调用
    private LocalDateTime createdAt;
}
```

**采集点**：在 `NvcAgentChatService.chat()` 和 `chatStream()` 中解析 LLM 响应的 `usage` 字段。

**LLM 响应中的 usage 格式**（OpenAI 兼容）：
```json
{
  "usage": {
    "prompt_tokens": 1234,
    "completion_tokens": 567,
    "total_tokens": 1801
  }
}
```

**成本计算**（以 DashScope qwen-plus 为例）：
```
输入: ¥0.004 / 1K tokens
输出: ¥0.012 / 1K tokens
成本 = promptTokens * 0.004 / 1000 + completionTokens * 0.012 / 1000
```

**Dashboard 扩展**：
- 总 Token 消耗（按天/周/月）
- 各 Agent 场景的 Token 消耗占比（饼图）
- 单次练习平均成本
- 每用户每日 Token 上限（防刷，可配置）

### 1.2 Prompt 版本管理与 A/B 测试 ⭐⭐⭐

#### 问题

Agent 的 System Prompt 存在 `NvcAgentConfigEntity.systemPrompt` 中，改了就覆盖，没有版本历史。无法回答：
- 这个 Prompt 是什么时候改的？
- 改了之后效果变好了还是变差了？
- 能不能同时测试两个 Prompt 版本？

#### 方案

**新增实体**：

```java
@Entity
@Table(name = "nvc_agent_prompt_version")
public class NvcAgentPromptVersionEntity {
    @Id @GeneratedValue
    private Long id;
    @Enumerated(EnumType.STRING)
    private NvcAgentScene agentScene;
    private Integer version;
    @Column(columnDefinition = "TEXT")
    private String systemPrompt;
    private Boolean isActive;           // 当前激活版本
    private String changeNote;          // 变更说明

    // A/B 测试指标（定期聚合）
    private Integer totalCalls;         // 调用次数
    private Double avgEvaluationScore;  // 平均评估分数
    private Double avgTokenUsage;       // 平均 Token 消耗
    private Double avgLatencyMs;        // 平均延迟
    private Double userSatisfactionRate; // 用户好评率（需要 1.6 的反馈数据）

    private LocalDateTime createdAt;
    private LocalDateTime activatedAt;
}
```

**版本切换流程**：
1. 创建新版本（`version = max + 1`，`isActive = false`）
2. 灰度发布：按 userId 取模分配版本（10% 流量走新版本）
3. 观察指标：对比两个版本的评估分数、Token 消耗、用户反馈
4. 全量发布：新版本指标更优则设为 `isActive = true`，旧版本 `isActive = false`

**A/B 测试路由**（在 `NvcAgentChatService` 中）：
```java
private NvcAgentPromptVersionEntity selectVersion(NvcAgentScene scene, Long userId) {
    List<NvcAgentPromptVersionEntity> versions = versionRepository
        .findByAgentSceneAndActive(scene, true);
    if (versions.size() <= 1) {
        return versions.get(0); // 无灰度，返回唯一激活版本
    }
    // A/B 路由：userId 取模
    int bucket = (int) (userId % 100);
    int splitPoint = versions.get(0).getTrafficPercentage(); // 如 90
    return bucket < splitPoint ? versions.get(0) : versions.get(1);
}
```

### 1.4 LLM 响应质量监控 ⭐⭐

#### 问题

LLM 调用的延迟、成功率、重试率没有监控，出了问题只能靠用户投诉发现。

#### 方案

**监控指标**：

| 指标 | 类型 | 说明 |
|------|------|------|
| `llm_call_latency_ms` | Histogram | LLM 调用延迟分布 |
| `llm_call_total` | Counter | LLM 调用总数（按 agent_scene、status 标签） |
| `llm_call_error_total` | Counter | LLM 调用失败数 |
| `llm_call_retry_total` | Counter | LLM 调用重试数 |
| `tool_call_total` | Counter | 工具调用次数（按 tool_name 标签） |
| `tool_call_latency_ms` | Histogram | 工具调用延迟 |
| `evaluation_score` | Summary | 评估分数分布（按 dimension 标签） |

**实现方式**：使用 Micrometer（Spring Boot 4.0 内置）：
```java
@Component
public class LlmMetrics {
    private final MeterRegistry registry;

    public void recordLlmCall(String scene, long latencyMs, boolean success) {
        registry.timer("llm.call.latency", "scene", scene)
            .record(latencyMs, TimeUnit.MILLISECONDS);
        registry.counter("llm.call.total", "scene", scene, "success", String.valueOf(success))
            .increment();
    }
}
```

## 二、语义缓存 — 降低 LLM 调用成本 ⭐⭐⭐

### 问题

用户问"什么是观察和评论的区别？"，每次都要调 LLM，但答案其实高度相似。知识问答类问题的重复率可达 30-50%。

### 方案

**核心思路**：用 Embedding 相似度判断是否命中缓存。

```
用户消息
    ↓
EmbeddingModel.embed(userMessage) → 查询向量
    ↓
pgvector 相似度搜索（nvc_semantic_cache 表）
    ├── cosine > 0.95 → 命中缓存，直接返回
    └── cosine ≤ 0.95 → 调用 LLM → 存入缓存
```

**新增表**：

```sql
CREATE TABLE nvc_semantic_cache (
    id BIGSERIAL PRIMARY KEY,
    query_text TEXT NOT NULL,
    query_embedding vector(1536) NOT NULL,
    response TEXT NOT NULL,
    agent_scene VARCHAR(50),
    hit_count INT DEFAULT 0,
    last_hit_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT NOW(),
    expires_at TIMESTAMP  -- 可选：缓存过期时间
);

CREATE INDEX ON nvc_semantic_cache USING hnsw (query_embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);
```

**缓存策略**：
- **写入条件**：只有"知识问答"类场景缓存（DIALOGUE_GUIDE、NVC_KNOWLEDGE_ADVISOR），角色扮演类不缓存（DIFFICULT_PARTNER）
- **命中阈值**：cosine similarity > 0.95（可通过配置调整）
- **过期策略**：30 天未命中自动清理
- **手动失效**：Prompt 更新时可选择清空缓存

**文件清单**：

| 文件 | 操作 | 说明 |
|------|------|------|
| `modules/nvcpractice/service/NvcSemanticCacheService.java` | 新增 | 语义缓存服务（查询 + 写入 + 清理） |
| `modules/nvcpractice/model/NvcSemanticCacheEntity.java` | 新增 | 缓存实体 |
| `modules/nvcpractice/repository/NvcSemanticCacheRepository.java` | 新增 | 缓存 Repository（含向量查询） |
| `modules/nvcpractice/service/NvcAgentChatService.java` | 修改 | 在调用 LLM 前检查缓存 |
| `docker/postgres/init.sql` | 修改 | 建表 + HNSW 索引 |

**工作量**：3 天

**简历亮点**："实现基于向量相似度的语义缓存，减少 30-50% 的 LLM 调用，响应时间从 2-3s 降至 100ms"

### 3.1 结构化日志 + 链路追踪 ⭐⭐⭐

#### 问题

当前日志是非结构化的纯文本：
```
2026-07-24 10:30:15 INFO NvcAgentOrchestrator - Agent decision: scene=DIALOGUE_GUIDE, reason=用户首次发言
```

无法聚合分析，无法跨服务追踪。

#### 方案

**1. 结构化日志（JSON 格式）**：

```xml
<!-- logback-spring.xml -->
<appender name="JSON" class="ch.qos.logback.core.ConsoleAppender">
    <encoder class="net.logstash.logback.encoder.LogstashEncoder">
        <customFields>{"service":"nvc-backend"}</customFields>
    </encoder>
</appender>
```

输出示例：
```json
{
  "timestamp": "2026-07-24T10:30:15.123Z",
  "level": "INFO",
  "logger": "nvc.guide.modules.nvcpractice.service.NvcAgentOrchestrator",
  "message": "Agent decision",
  "service": "nvc-backend",
  "traceId": "abc123",
  "spanId": "def456",
  "sessionId": 42,
  "scene": "DIALOGUE_GUIDE",
  "reason": "用户首次发言"
}
```

**2. 链路追踪（Micrometer Tracing）**：

Spring Boot 4.0 内置 Micrometer Tracing，自动为每个请求生成 traceId。

```java
@Service
public class NvcAgentOrchestrator {
    private final Tracer tracer;

    public AgentDecision decideNextAgent(PracticeContext context) {
        Span span = tracer.nextSpan().name("decideNextAgent").start();
        try (Tracer.SpanInScope ws = tracer.withSpan(span)) {
            span.tag("session.id", String.valueOf(context.getSession().getId()));
            span.tag("practice.mode", context.getSession().getPracticeMode().name());
            // ... 业务逻辑
            span.tag("decision.scene", decision.scene().name());
            return decision;
        } finally {
            span.end();
        }
    }
}
```

```java
@Component

    @Override

    }
}
```

```yaml
spring:
  flyway:
```

### 4.2 索引优化 ⭐⭐

#### 问题

当前缺少关键索引，数据量大后查询变慢。

#### 方案

```sql
-- pgvector HNSW 索引（向量检索性能提升 10-100 倍）
CREATE INDEX IF NOT EXISTS idx_vector_store_embedding
    ON vector_store USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

-- 对话历史查询（按会话 + 顺序）
CREATE INDEX IF NOT EXISTS idx_practice_message_session_seq
    ON nvc_practice_message (session_id, sequence_num);

-- 评估历史查询（按会话 + 时间）
CREATE INDEX IF NOT EXISTS idx_evaluation_session_time
    ON nvc_evaluation (session_id, created_at);

-- 能力分数查询（按用户 + 维度）
CREATE INDEX IF NOT EXISTS idx_ability_score_user_dim
    ON nvc_user_ability_score (user_id, dimension);

-- Token 用量统计（按用户 + 时间）
CREATE INDEX IF NOT EXISTS idx_token_usage_user_time
    ON nvc_token_usage (user_id, created_at);

-- 语义缓存向量检索
CREATE INDEX IF NOT EXISTS idx_semantic_cache_embedding
    ON nvc_semantic_cache USING hnsw (query_embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);
```

**工作量**：0.5 天

**效果**：pgvector HNSW 索引对 10 万条向量的检索时间从 500ms 降至 5ms。

### 4.3 读写分离（远期）⭐

当用户量上来后（日活 1000+），可考虑：

**方案**：
- 主库：写操作（保存消息、评估结果）
- 从库：读操作（查询历史、仪表盘统计）
- Spring Boot + `@Transactional(readOnly=true)` 自动路由

```java
@Configuration
public class DataSourceConfig {
    @Bean
    @Primary
    public DataSource primaryDataSource() {
        // 主库（写）
        return DataSourceBuilder.create().url("jdbc:postgresql://master:5432/nvc").build();
    }

    @Bean
    public DataSource replicaDataSource() {
        // 从库（读）
        return DataSourceBuilder.create().url("jdbc:postgresql://replica:5432/nvc").build();
    }
}
```

**工作量**：2 天（含配置和测试）

**适用场景**：日活 1000+ 时考虑，当前不需要。

```java
@Component
public class PromptInjectionDetector {
    private static final List<Pattern> INJECTION_PATTERNS = List.of(
        Pattern.compile("(?i)ignore\\s+(all\\s+)?previous\\s+instructions"),
        Pattern.compile("(?i)you\\s+are\\s+now\\s+a"),
        Pattern.compile("(?i)system\\s*:\\s*"),
        Pattern.compile("(?i)forget\\s+(everything|all)"),
        Pattern.compile("(?i)override\\s+(your|system)\\s+instructions")
    );

    public boolean detect(String input) {
        return INJECTION_PATTERNS.stream()
            .anyMatch(p -> p.matcher(input).find());
    }
}
```

### 6.1 领域事件（Domain Events）⭐⭐

```java
    private final Long sessionId;

}

// 2. 发布事件
@Service
public class NvcPracticeSessionService {

    public CompleteResult completeAndEvaluate(Long sessionId) {
        // ... 完成逻辑
        eventPublisher.publishEvent(new PracticeCompletedEvent(sessionId, userId));
        return result;
    }
}

// 3. 监听事件（各模块独立监听）
@Component
public class NvcEvaluationEventListener {

    @Async

}

@Component
public class NvcWikiEventListener {
```

### 6.2 多模型 Fallback ⭐

```yaml
      - provider: deepseek

        model: llama3
```

```java
// LlmProviderRegistry 中增加 fallback 方法
public ChatClient getChatClientWithFallback(String primaryProvider) {
    for (String provider : fallbackChain) {
        try {
            ChatClient client = getChatClient(provider);
            // 发送简单测试
            return client;
        } catch (Exception e) {
            log.warn("Provider {} failed, trying next: {}", provider, e.getMessage());
        }
    }
    throw new BusinessException(ErrorCode.ALL_PROVIDERS_FAILED, "所有 LLM 提供商不可用");
}
```

### 6.3 Prompt 模板引擎 ⭐

```java
String prompt = "你是NVC教练。" + "\n\n[用户档案]\n" + profileSummary
```

```
<scenario>
<endif>

<if(ragContext)>

<endif>
```

```java
@Service
public class NvcPromptRenderer {

    public String render(String templateName, Map<String, Object> context) {

        return template.render();
    }
}
```
