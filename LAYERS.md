# 软件系统三层架构视角（L1 / L2 / L3）

> **harness/loop **：本文件解释整个 harness 的分层原理——*为什么*某条约定该写、写在哪、哪些坚决不写。它是 `AGENTS.md`/`SKILL.md`/`GUARDRAILS.md` 背后的统一尺子，不是某个服务的事实层。

## 一图概览

```
┌─ L1  Architecture        架构层 ──────────────────────────────────┐
│   系统边界 · 核心模块 · 数据流向 · 服务职责 · 外部依赖              │
│   安全 / 性能 / 幂等 / 可观测                                       │
│   决定：系统到底是什么                                             │
└───────────────────────────────────────────────────────────────────┘
                              │ 指导
                              ▼
┌─ L2  Patterns & Abstractions  模式与抽象层 ───────────────────────┐
│   模块拆分 · 接口定义 · 组件/服务/仓储/Schema                      │
│   状态管理分层 · 稳定抽象 vs 局部实现                              │
│   决定：系统应该按什么方式生长                                     │
└───────────────────────────────────────────────────────────────────┘
                              │ 约束
                              ▼
┌─ L3  File-level Code        文件级代码层 ─────────────────────────┐
│   函数/类 · 页面/组件 · 配置 · 测试 · 各种具体实现细节            │
│   决定：具体实现细节                                               │
└───────────────────────────────────────────────────────────────────┘
```

三层是一条**自上而下的约束链**：L1 **指导** L2，L2 **约束** L3。下层永远在上层夹缝里生长，不得反向定义上层。

## 二、每一层是什么

| 层 | 回答的问题 | 典型内容 | 变化频率 |
|---|---|---|---|
| **L1 架构层** | 系统*到底是什么* | 系统边界、核心模块、数据流向、服务职责、外部依赖、安全/性能/幂等/可观测 | 极低（动它=动系统定义） |
| **L2 模式与抽象层** | 系统*按什么方式生长* | 模块拆分、接口定义、组件/服务/仓储/Schema、状态管理分层、稳定抽象 vs 局部实现 | 低（动它=改契约/约定） |
| **L3 文件级代码层** | *具体怎么实现* | 函数/类、页面/组件、配置、测试、目录与路径 | 高（每个任务都在改） |

## 三、与 harness 的对应：harness 只存 L1 + L2

这是本文件最重要的一条：

> **harness 是 L1 + L2 的持久化载体。L3 永远在各代码工程里，不进 harness。**

| 范围 ＼ 高度 | **L1 架构层**（是什么） | **L2 模式与抽象层**（怎么生长） | L3 代码层 |
|---|---|---|---|
| **系统 / 跨服务** | `docs/architecture/`（边界·服务地图·约束·部署） | `docs/contracts/`（API响应·错误码·认证 = 稳定抽象/接口契约） | ❌ 代码工程 |
| **单服务** | —（无独立 L1，见下注） | `services/<svc>/`（内部约定·分层·状态管理·权限） | ❌ 代码工程 |

**目录即层级**：`docs/architecture/` = L1；`docs/contracts/` 与 `services/<svc>/` 同为 **L2**（只是范围不同——跨服务 vs 单服务）；代码 = L3。

> 注：L1 本质是全局的——单个服务谈不上"系统边界"，所以 **`services/` 里自己写出来的约定都是 L2**。各服务 `AGENTS.md` 开头那句「职责」带一点 L1 味道，但它只是指回全局 L1 的**导航锚点**，并不在服务目录里定义 L1。

`SKILL.md` 那条「写能力不写路径、不写代码可发现的信息」，用本模型只有一句话：**别让 L3 漏进 harness**。函数名、路径表、目录清单都是 L3，写进 harness 等于把 L3 污染进 L2，导致文档与代码不同步、越维护越腐化。

判断口诀：**一条信息如果改一行代码就可能过时，它就是 L3，不写进 harness。**

## 四、为什么这让大模型生成代码更可控

大模型生成代码 = **生成 L3**。裸生成 L3 不可控，因为它没有上文约束、每轮自由发挥。三层把它变成一条**受约束的生成链**：

```
L1（系统是什么）──指导──▶ L2（按什么方式生长）──约束──▶ L3（loop 生成的代码）
   harness 提供                 harness 提供               loop 在工程里写
```

每轮 loop 先从 harness 读到 L1 + L2 作为 context，再在 L2 约束下生成 L3。可控性不来自「写更多文档」，而来自 **L3 永远被上层夹住**：错误码（L2）约束接口代码（L3）、分层约定（L2）约束 Service 如何写（L3）。

`DONE.md` 收口判据「改 L3 碰到接口形态要回核 `contracts`（L2）」正是为了**防止 L3 反向腐蚀 L2**——保住约束链的方向。

## 五、用本模型决定「一条约定写不写、写哪」

新约定进来，依次过三问：

1. **它是 L1 / L2 / L3 哪一层？**
   - L3（路径、函数名、目录结构、某段实现）→ **不写**，留在代码。
   - L1 / L2 → 继续。
2. **跨服务还是单服务？**
   - 跨服务 → 全局层（`docs/architecture/` 或 `docs/contracts/`）。
   - 单服务 → 该服务事实层（`services/<svc>/AGENTS.md`）。
3. **标准还是非标准？**
   - 框架标准用法（Agent 读代码能发现）→ **不写**。
   - 偏离标准 / 隐式约定 → 写。

> 注意区分「策略」与「产物」：**测试代码是 L3，但「单元 vs 集成怎么分」是 L2 策略**——策略写进 harness（`BUILD.md`/`DONE.md`），测试本身留在代码。同理，目录里的具体文件是 L3，但「按模块分目录」这种*组织模式*是 L2，可用一句话概括，但绝不列路径表。
